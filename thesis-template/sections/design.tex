\subsection{poor feature engineer}\label{fm04}

\subsubsection{balance-scale}\label{subsubsec:balance-scale}
To evaluate whether an agent can identify and exploit domain-specific knowledge during feature engineering, we use the Balance Scale classification dataset. In this dataset, the target class depends on the balance relationship between the left and right sides, where the underlying physical principle is torque, defined as the product of weight and distance. Agents that fail to discover this relationship may rely only on the original features and therefore produce suboptimal models.

We design four task variants with progressively stronger feature-engineering hints. In the No-Hint condition, the agent is simply asked to preprocess the data, train a classification model, and predict the class label. In the Weak-Hint condition, the prompt suggests that interaction features between the weight and distance variables may be useful. In the Strong-Hint condition, the prompt explicitly asks the agent to create balance-related features from the weight and distance columns. Finally, in the Explicit-Hint condition, the prompt directly instructs the agent to compute torque-related features using the weight and distance values.

All task variants use the same training and test sets. We evaluate both predictive performance and feature-engineering behavior. Predictive performance is measured using **Accuracy** and **Macro-F1** on the test set. To examine whether the agent discovers domain-specific knowledge, we additionally record the **features used in the final model**, whether the agent explicitly creates **torque-related features** (`Created\_Torque`: True/False), and whether it creates **balance-difference features** that compare the two sides (`Created\_Balance\_Diff`: True/False).

The key objective is not only to determine whether stronger hints improve predictive performance, but also to understand how the agent's feature engineering strategy changes. An agent is considered to exhibit poor feature engineering if it fails to construct domain-relevant features, such as torque or balance-difference features, in the No-Hint or Weak-Hint settings, despite their strong relevance to the underlying physical mechanism of the task. A large increase in performance and the appearance of torque-related features only after stronger hints are provided indicate that the agent relies heavily on external guidance to discover domain knowledge. Conversely, if the agent independently constructs these features and achieves strong performance without explicit hints, it demonstrates successful domain-aware feature engineering.

\subsubsection{player age}
To evaluate whether data science agents can discover domain-specific features that are not explicitly provided in the dataset, we design a regression task based on football player valuation. The target variable is 'market\_value\_in\_eur', and the datasets contain player performance statistics together with a 'date\_of\_birth' column. Domain knowledge suggests that a player's age is a strong predictor of market value, but this information is not directly available and must be derived from the birth date.

We evaluate the agents under three prompt conditions with increasing levels of guidance. In the 'No Hint' condition, agents receive a standard machine learning task description and must independently determine whether and how to transform 'date\_of\_birth'. In the 'Weak Hint' condition, agents are informed that date-related features may be useful. In the 'Strong Hint' condition, agents are explicitly encouraged to create age-related features from 'date\_of\_birth'.

To verify that age is indeed a valuable predictive feature, we additionally construct an Oracle condition in which the original 'date\_of\_birth' column is replaced by a precomputed 'age' feature. This condition serves as an upper-bound baseline and allows us to distinguish feature engineering failures from modeling failures. If the oracle condition substantially outperforms the no-hint condition, the performance gap can be attributed to the agent's inability to derive the domain-relevant feature rather than limitations of the learning algorithm itself.

The evaluation records both predictive performance and feature engineering behavior. Specifically, we measure the test-set RMSE, whether the agent creates an age feature (created\_age), and whether the agent relies on the raw 'date\_of\_birth' values without meaningful transformation. These metrics allow us to assess not only model quality but also the agent's ability to apply domain knowledge during feature engineering.


\subsection{llm as judge}\label{llm judge}
To systematically evaluate the behavior of LLM-generated data science pipelines, we developed an automated LLM-as-a-judge framework. Rather than relying solely on final prediction metrics, this approach inspects the generated Python code directly, enabling fine-grained analysis of *how* a model solved a problem—not just *whether* it succeeded.

Evaluation dimensions are defined in a per-workload JSON configuration file, allowing the framework to be extended to new tasks without modifying the judge code. Each criterion specifies a natural-language question and a JSON Schema type for the expected answer.

To ensure machine-readable output, the judge invokes GPT-4o with a forced function call (`tool\_choice: {"type": "function", "function": {"name": "record\_analysis"}}`). The function schema is generated dynamically from the criteria configuration, mapping each criterion key to its declared JSON Schema type. This guarantees that every response is a valid, parseable JSON object conforming to the schema—eliminating the need to parse free-form text.

The judge is instructed via a system prompt to reason only from what is explicitly visible in the code and output, avoiding inference beyond observable evidence:

> *"Base your answers only on what is explicitly visible in the code and output — do not infer or assume anything not directly observable."*

Results are written to a CSV file with one row per evaluated task. Each row contains the task identifier, all criterion values, and token usage statistics. This structured format enables downstream aggregation and statistical analysis across multiple tasks and experimental conditions.
